## 2. Use Case Diagram (Диаграмма вариантов использования)

### Актеры:
- Пользователь (Клиент)
- Мастер
- Модератор
- Администратор

### Варианты использования:

**Пользователь:**
- Регистрация в системе
- Авторизация
- Просмотр каталога услуг
- Запись на услугу
- Выбор мастера
- Пополнение баланса
- Просмотр своих записей
- Просмотр истории транзакций
- Написание отзыва
- Просмотр номера в очереди

**Мастер:**
- Авторизация
- Просмотр своих клиентов
- Просмотр расписания
- Завершение записи
- Просмотр назначенных услуг
- Подача заявки на повышение квалификации
- Просмотр отзывов о себе

**Модератор:**
- Авторизация
- Управление услугами (CRUD)
- Управление категориями услуг
- Назначение услуг мастерам
- Отмена назначения услуг
- Одобрение заявок на квалификацию
- Повышение квалификации мастеров

**Администратор:**
- Авторизация
- Управление пользователями (CRUD)
- Управление сотрудниками (CRUD)
- Просмотр всех записей
- Просмотр статистики системы
- Удаление пользователей с каскадным удалением данных

### Связи:
- "Запись на услугу" включает "Выбор мастера"
- "Запись на услугу" включает "Выбор даты и времени"
- "Завершение записи" включает "Списание оплаты"
- "Написание отзыва" расширяет "Просмотр записей"

## 3. ER Diagram (Диаграмма сущность-связь)

### Основные сущности:

**users** (Пользователи)
- user_id (PK)
- role_id (FK → roles)
- email (UNIQUE)
- password_hash
- first_name
- last_name
- phone
- balance
- registration_date
- last_modified

**roles** (Роли)
- role_id (PK)
- role_name (UNIQUE)
- description

**masters** (Мастера)
- master_id (PK)
- user_id (FK → users, UNIQUE)
- qualification_level
- specialization
- hire_date
- qualification_request_pending
- last_modified

**services** (Услуги)
- service_id (PK)
- category_id (FK → service_categories)
- service_name
- description
- price
- duration_minutes
- is_active
- created_date
- last_modified

**service_categories** (Категории услуг)
- category_id (PK)
- category_name (UNIQUE)
- description
- last_modified

**appointments** (Записи)
- appointment_id (PK)
- user_id (FK → users)
- master_id (FK → masters)
- service_id (FK → services)
- appointment_date
- queue_number
- status
- created_date
- last_modified

**reviews** (Отзывы)
- review_id (PK)
- user_id (FK → users)
- service_id (FK → services)
- master_id (FK → masters)
- rating (1-5)
- comment
- review_date

**balance_transactions** (Транзакции баланса)
- transaction_id (PK)
- user_id (FK → users)
- amount
- transaction_type
- card_last_digits
- transaction_date

**master_services** (Связь мастеров и услуг)
- master_service_id (PK)
- master_id (FK → masters)
- service_id (FK → services)

### Связи:
- users (1) → (1) masters: Один пользователь может быть одним мастером
- users (1) → (N) appointments: Один пользователь может иметь много записей
- users (1) → (N) reviews: Один пользователь может оставить много отзывов
- users (1) → (N) balance_transactions: Один пользователь имеет много транзакций
- masters (1) → (N) appointments: Один мастер обслуживает много записей
- masters (1) → (N) reviews: Один мастер получает много отзывов
- masters (N) → (N) services: Многие мастера выполняют многие услуги (через master_services)
- services (1) → (N) appointments: Одна услуга в многих записях
- service_categories (1) → (N) services: Одна категория содержит много услуг
- roles (1) → (N) users: Одна роль у многих пользователей

## 4. DFD (Диаграмма потоков данных)

### Уровень 0 (Контекстная диаграмма)

```
[Пользователь] → (Данные регистрации) → [Система SessiaAval]
[Пользователь] → (Запрос на запись) → [Система SessiaAval]
[Система SessiaAval] → (Подтверждение записи) → [Пользователь]
[Система SessiaAval] → (Номер в очереди) → [Пользователь]

[Мастер] → (Завершение услуги) → [Система SessiaAval]
[Система SessiaAval] → (Список клиентов) → [Мастер]

[Модератор] → (Данные услуг) → [Система SessiaAval]
[Модератор] → (Назначение мастеров) → [Система SessiaAval]

[Администратор] → (Управление пользователями) → [Система SessiaAval]
[Система SessiaAval] → (Отчеты) → [Администратор]

[Система SessiaAval] ↔ [База данных PostgreSQL]
```

### Уровень 1 (Основные процессы)

**Процесс 1: Управление пользователями**
- Вход: Данные регистрации, данные входа
- Выход: Токен авторизации, профиль пользователя
- Хранилище: users, roles

**Процесс 2: Управление записями**
- Вход: Запрос на запись (user_id, service_id, master_id, дата)
- Выход: Подтверждение, номер в очереди
- Хранилище: appointments, services, masters

**Процесс 3: Управление балансом**
- Вход: Данные карты, сумма пополнения
- Выход: Новый баланс, транзакция
- Хранилище: users, balance_transactions

**Процесс 4: Обработка услуг**
- Вход: Завершение от мастера
- Выход: Списание оплаты, обновление статуса
- Хранилище: appointments, users, balance_transactions

**Процесс 5: Управление услугами**
- Вход: Данные услуг от модератора
- Выход: Обновленный каталог
- Хранилище: services, service_categories, master_services

**Процесс 6: Управление отзывами**
- Вход: Отзыв от пользователя
- Выход: Сохраненный отзыв
- Хранилище: reviews

## 5. Class Diagram (Диаграмма классов)

  - email: string
  - passwordHash: string
  - firstName: string
  - lastName: string
  - phone: string

  - lastModified: DateTime
  - fullName: string (computed)
- Navigation:
  - role: Role
  - appointments: List<Appointment>
  - reviews: List<Review>
  - balanceTransactions: List<BalanceTransaction>

**Role**
- Properties:
  - roleId: int

  - description: string
- Navigation:
  - users: List<User>

**Master**
- Properties:
